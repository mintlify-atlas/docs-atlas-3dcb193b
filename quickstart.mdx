## Quick Setup

<Steps>
  <Step title="Clone and Install">
    Get the source code and install dependencies:

    ```bash
    git clone <repository-url>
    cd firedancer-frontend
    npm install
    ```

    <Accordion title="Installation takes too long?">
      The first `npm install` downloads hundreds of packages. This is normal and may take 2-5 minutes depending on your connection speed. Subsequent installs use npm's cache and are much faster.
    </Accordion>
  </Step>

  <Step title="Configure WebSocket Connection">
    Create your development environment file:

    ```bash
    cp .env.example .env.development
    ```

    Edit `.env.development` to point to your validator:

    ```bash .env.development
    VITE_WEBSOCKET_URL=ws://localhost:7999/websocket
    VITE_VALIDATOR_CLIENT=Firedancer
    ```

    <ParamField path="VITE_WEBSOCKET_URL" type="string" required>
      WebSocket endpoint of your validator. Common configurations:
      
      - **Local validator**: `ws://localhost:7999/websocket`
      - **Remote validator**: `ws://192.168.1.100:7999/websocket`
      - **Production** (auto-detect): Leave empty
    </ParamField>

    <ParamField path="VITE_VALIDATOR_CLIENT" type="enum" required>
      Set to `Firedancer` or `Frankendancer` to match your validator type.
    </ParamField>

    <Warning>
    Make sure your validator's WebSocket server is running and accessible at the specified URL. The dashboard cannot function without an active WebSocket connection.
    </Warning>
  </Step>

  <Step title="Start the Development Server">
    Launch the Vite dev server:

    ```bash
    npm run dev
    ```

    You should see output like:

    ```
    VITE v5.4.8  ready in 523 ms

      ➜  Local:   http://localhost:5173/
      ➜  Network: http://0.0.0.0:5173/
      ➜  press h to show help
    ```

    <Check>
    The dashboard is now running! Open http://localhost:5173/ in your browser.
    </Check>
  </Step>
</Steps>

## Running the Dev Server

The development server provides hot module replacement (HMR) for instant updates as you edit code.

### Development Scripts

```json package.json
{
  "scripts": {
    "dev": "NODE_OPTIONS=--max-http-header-size=64000 vite",
    "build": "tsc && vite build",
    "build:fr": "tsc && VITE_VALIDATOR_CLIENT=Frankendancer vite build",
    "build:fd": "tsc && VITE_VALIDATOR_CLIENT=Firedancer vite build"
  }
}
```

<CodeGroup>
```bash Standard Dev Server
npm run dev
```

```bash Preview Production Build
npm run preview
```

```bash Build for Production
npm run build
```
</CodeGroup>

### Dev Server Features

- **Hot Module Replacement**: Changes reflect instantly without page refresh
- **TypeScript Type Checking**: Real-time type errors in the terminal
- **Automatic Port Selection**: If port 5173 is busy, Vite picks the next available port
- **Network Access**: Access from other devices on your network via the Network URL

<Accordion title="Change the default port">
If port 5173 conflicts with another service, modify `vite.config.ts`:

```ts vite.config.ts
export default defineConfig({
  server: {
    port: 3000  // or any port you prefer
  }
})
```
</Accordion>

## Connecting to Your Validator

The dashboard establishes a WebSocket connection on load via the `ConnectionProvider` component.

### Connection Flow

```tsx src/api/ws/ConnectionProvider.tsx
const VITE_WEBSOCKET_URL = import.meta.env.PROD
  ? null
  : (import.meta.env.VITE_WEBSOCKET_URL as string)?.trim();

// connect to current host via websocket
const websocketUrl = VITE_WEBSOCKET_URL
  ? VITE_WEBSOCKET_URL
  : `${window.location.protocol.startsWith("https") ? "wss" : "ws"}://${window.location.hostname}:${window.location.port}/websocket`;
```

**Development mode**: Uses `VITE_WEBSOCKET_URL` from `.env.development`  
**Production mode**: Auto-detects from the browser's current location

### Connection States

The dashboard displays different states based on WebSocket status:

| State | Description | UI Indicator |
|-------|-------------|--------------|
| **Connecting** | Establishing WebSocket connection | Startup progress screen |
| **Connected** | Active data stream from validator | Green status indicator |
| **Reconnecting** | Temporary disconnection, retrying | Yellow status indicator |
| **Disconnected** | No connection to validator | Red status indicator with error |

### Troubleshooting Connection Issues

<AccordionGroup>
  <Accordion title="Connection refused error">
    **Symptom**: WebSocket error in browser console: `WebSocket connection failed`

    **Solutions**:
    1. Verify your validator is running: `systemctl status firedancer`
    2. Check the WebSocket URL in `.env.development` matches your validator's config
    3. Ensure firewall allows the connection: `sudo ufw allow 7999/tcp`
    4. Test the endpoint directly: `curl -i -N -H "Connection: Upgrade" -H "Upgrade: websocket" http://localhost:7999/websocket`
  </Accordion>

  <Accordion title="Stuck on startup progress screen">
    **Symptom**: Dashboard shows loading screen indefinitely

    **Causes**:
    - Validator is still catching up
    - WebSocket connected but not receiving data
    - CORS policy blocking connection

    **Solutions**:
    1. Check browser DevTools Console for errors
    2. Verify validator logs show WebSocket server running
    3. Try hard refresh: `Ctrl+Shift+R` (Windows/Linux) or `Cmd+Shift+R` (Mac)
    4. Clear browser cache and restart dev server
  </Accordion>

  <Accordion title="Auto-disconnect after a few seconds">
    **Symptom**: Connects successfully but disconnects shortly after

    **Possible causes**:
    - Validator WebSocket timeout too aggressive
    - Network proxy interfering with connection
    - Validator resource constraints

    **Solutions**:
    1. Check validator resource usage (CPU, memory)
    2. Review validator logs for WebSocket errors
    3. Try connecting from the same machine as the validator: `ws://localhost:7999/websocket`
  </Accordion>
</AccordionGroup>

- Execution timings
- Scheduler statistics

See [Slot Details](/features/slot-details) for complete documentation.

### Monitoring Tile Performance

The Live Tile Metrics table shows real-time performance for Firedancer's tile-based architecture:

```tsx src/features/Overview/LiveTileMetrics/index.tsx
function TableRow({ tile, liveTileMetrics, idx }: TableRowProps) {
  const alive = liveTileMetrics.alive[idx];
  const cpu = liveTileMetrics.last_cpu[idx];
  const inBackpressure = liveTileMetrics.in_backp[idx];
  
  // Tile timer percentages
  const timers = liveTileMetrics.timers[idx];
  const hKeepPct = timers[0] + timers[1] + timers[2];
  const waitPct = timers[6];
  const backpPct = timers[5];
  const workPct = timers[3] + timers[4] + timers[7];

  // ...
}
```

**Key metrics to watch**:
- **Utilization**: Should be below 80% for healthy operation
- **Backpressure %**: High values (>10%) indicate bottlenecks
- **Involuntary Context Switches**: Rapid increases suggest CPU contention
- **Status**: Any "Dead" tiles require immediate attention

### Network Traffic Analysis

The Live Network Metrics panel breaks down ingress/egress traffic by protocol:

- **QUIC**: Transaction data from clients and other validators
- **Gossip**: Cluster coordination and peer discovery
- **Repair**: Shred recovery operations

Monitor these to identify:
- Bandwidth saturation
- Unusual traffic patterns
- Protocol-specific issues

### Gossip Network Health

Navigate to **Gossip** (`src/routes/gossip.tsx`) to view:

- Connected peer list with stake information
- Push/pull message statistics
- Traffic throughput per peer
- Message success/failure rates

This is crucial for diagnosing network connectivity issues.
