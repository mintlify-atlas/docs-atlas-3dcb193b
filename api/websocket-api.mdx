# WebSocket API

The Firedancer Frontend connects to the validator backend via WebSocket to receive real-time updates about validator performance, slot processing, peer information, and more.

## Connection Management

### Connection States

The WebSocket connection can be in one of three states:

```typescript
export enum SocketState {
  Disconnected = "disconnected",
  Connecting = "connecting",
  Connected = "connected",
}
```

### Connection Configuration

The WebSocket connection supports optional Zstandard compression for reduced bandwidth:

```typescript
const VITE_WEBSOCKET_COMPRESS = import.meta.env.VITE_WEBSOCKET_COMPRESS?.trim();

ws = new WebSocket(
  url,
  VITE_WEBSOCKET_COMPRESS === "false" ? undefined : ["compress-zstd"]
);
```

### Automatic Reconnection

The connection automatically reconnects with a 3-second delay if disconnected:

```typescript
const RECONNECT_DELAY = 3_000;

ws.onclose = function onclose() {
  logDebug("WS", `Disconnected, reconnecting in ${RECONNECT_DELAY}ms`);
  onConnectionStatusChanged({ socketState: SocketState.Disconnected });
  clearTimeout(reconnectTimer);
  reconnectTimer = setTimeout(connect, RECONNECT_DELAY);
};
```

## Message Types

### Inbound Messages (Server → Client)

All messages from the server follow a topic-based structure with discriminated unions for type safety.

#### Message Structure

Every message contains a `topic` field that determines the message type:

```typescript
{
  topic: "summary" | "epoch" | "gossip" | "peers" | "slot" | "block_engine",
  key: string,
  value: unknown
}
```

## Message Handling

### Message Compression

Messages can be received in two formats:

1. **Plain JSON** (string)
2. **Zstandard-compressed binary** (ArrayBuffer)

```typescript
ws.onmessage = function onmessage(ev: MessageEvent<unknown>) {
  const message = ev.data;
  let json = undefined;
  
  if (typeof message === "string") {
    json = JSON.parse(message);
  } else if (message instanceof ArrayBuffer) {
    json = JSON.parse(
      decoder.decode(zstd.ZstdStream.decompress(new Uint8Array(message)))
    );
  }
  
  onMessage(json);
};
```

### Schema Validation

All incoming messages are validated against Zod schemas:

```typescript
try {
  const { topic } = topicSchema.parse(msg);
  
  if (topic === "summary") {
    const { key, value } = summarySchema.parse(msg);
    // Handle summary message
  } else if (topic === "slot") {
    const { key, value } = slotSchema.parse(msg);
    // Handle slot message
  }
  // ...
} catch (e) {
  if (e instanceof ZodError) {
    console.error("Schema validation failed", e.issues);
  }
}
```

### Outbound Messages (Client → Server)

The client can send messages to query specific data:

```typescript
type ClientMessage = unknown;

function sendMessage(message: ClientMessage) {
  if (ws && ws.readyState === WebSocket.OPEN) {
    ws.send(JSON.stringify(message));
  }
}
```

<Info>
The exact structure of client messages depends on the backend implementation. Common operations include querying historical slot data and requesting specific metrics.
</Info>

## Performance Optimizations

### Message Throttling

High-frequency updates are throttled to reduce React re-renders:

```typescript
const estimatedTpsDebounceMs = 1000;
const waterfallDebounceMs = 500;
const liveMetricsDebounceMs = 500;
const liveNetworkMetricsDebounceMs = 1000;
const liveTileMetricsDebounceMs = 1000;
```

### Peer Update Buffering

Peer updates are buffered and flushed periodically to reduce state updates:

```typescript
const peersBuffer = useRef(new Map<string, Peer>());
const removePeersBuffer = useRef(new Map<string, PeerRemove>());

const dbFlushBuffer = useDebouncedCallback(
  () => {
    updatePeers([...peersBuffer.current.values()]);
    removePeers([...removePeersBuffer.current.values()]);
    peersBuffer.current.clear();
    removePeersBuffer.current.clear();
  },
  1_000,
  { maxWait: 1_000 }
);
```
